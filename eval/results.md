# 면접 질문 품질 평가 (LLM-as-a-judge)

질문 생성 품질을 **정량적으로 측정**하고, 모델·프롬프트 변경의 효과를
변수별로 분리해 검증한 실험 기록.

## 실험 설계

- **방법**: LLM-as-a-judge — 생성과 다른 모델을 심사관으로 써서 질문을 채점
  - 생성: `gpt-5-mini` (V0만 `gpt-4o-mini`) / 심사관: `gpt-4o` (자기평가 편향 완화)
- **평가셋**: 이력서 3개(AI/ML·백엔드·데이터분석 신입) × 버전당 질문 9개 채점
- **지표(각 0/1)**: 난이도적합 · 직무관련 · 이력서근거 · 단일주제
- **변수 통제**: 한 번에 한 변수만 변경
  - `V0→V1`: 모델만 교체 (프롬프트 고정) → **모델 효과**
  - `V1→V2→V3`: 모델 고정, 프롬프트만 개선 → **프롬프트 효과**

| 버전 | 생성 모델 | 프롬프트 |
|---|---|---|
| V0 (원본) | gpt-4o-mini | 원본(약함) |
| V1 (모델교체) | gpt-5-mini | 원본(약함) |
| V2 (프롬프트) | gpt-5-mini | 직무·난이도·근거 |
| V3 (단일주제) | gpt-5-mini | + 단일주제 강제 |

## 결과

| 지표 | V0 (원본/4o-mini) | V1 (모델교체/5-mini) | V2 (프롬프트) | V3 (+단일주제) |
|---|---|---|---|---|
| 난이도적합 | 100% | 89% | 100% | 100% |
| 직무관련 | 89% | 100% | 100% | 100% |
| 이력서근거 | 100% | 89% | 100% | 100% |
| 단일주제 | 100% | 22% | 11% | 100% |
| **종합평균** | **97%** | **75%** | **78%** | **100%** |

## 핵심 발견

### 1. "최신 모델 = 무조건 개선"은 거짓 — 측정으로 반증
모델만 gpt-4o-mini → gpt-5-mini로 교체했더니(V0→V1) 종합 점수가
**97% → 75%로 오히려 하락**했다. 원인은 **단일주제 100% → 22%**:
gpt-5-mini가 더 똑똑해지면서 한 질문에 여러 하위질문을 욱여넣었다.

### 2. 모델 업그레이드는 프롬프트와 결합했을 때만 효과
프롬프트로 난이도·직무·근거를 보강(V2)해도 단일주제는 11%로 더 악화됐고,
**'단일 주제' 규칙을 명시적으로 강제(V3)한 뒤에야 100% 달성**.
즉 신모델의 깊이는 프롬프트 엔지니어링과 결합해야 비로소 품질로 이어졌다.

### 3. 회귀(regression)는 측정 없이는 안 보인다
프롬프트를 풍부하게 만들수록 단일주제가 더 나빠지는 회귀가 있었는데,
이는 눈으로는 못 잡고 지표로만 드러났다.

## 한계 (정직한 평가)

- **rubric이 '깊이/구체성'을 측정 못 함**: V0(97%)가 지표상 V3(100%)에
  근접하지만, 정성적으로 V0 질문은 일반적("Python 경험 이야기해주세요")이고
  V3는 구체적("임베딩 모델을 무엇을 왜 선택했나요")이다. → '구체성' 지표 추가 필요.
- **심사관 모델에 따라 절대 점수가 변동**: gpt-4o-mini 심사 시 난이도적합이
  더 낮게 나왔다. 단, **버전 간 상대 순위(V3>others)는 심사관과 무관하게 동일**해
  결론은 robust했다.
- **표본이 작음**(버전당 9문항): 방향성은 신뢰, 정밀 수치는 참고용. 이력서
  10~20개로 확장하면 안정성 향상.

## 결론

**채택: V3 (gpt-5-mini + 직무·난이도·근거·단일주제 프롬프트)** — 지표 종합 100%,
정성적으로도 가장 구체적·심층적. "측정으로 검증한 뒤 채택" 원칙에 따라 적용.

향후: `gpt-4o-mini + V3 프롬프트`(비용 최적 후보) 비교, '구체성' 지표 추가.

## 부록 — 정성 비교 (AI/ML 신입 이력서)

### V0 (원본/4o-mini) — 단일주제지만 일반적·얕음
1. Python을 사용하여 AI 프로젝트를 수행한 경험에 대해 이야기해 주실 수 있나요?
2. LangChain과 RAG를 활용한 간단한 예제를 설명해 주실 수 있나요?
3. CrewAI 멀티에이전트를 개발하면서 겪었던 어려움과 해결 방법에 대해 말씀해 주세요.

### V1 (모델교체/5-mini) — 깊지만 한 질문에 여러 개를 묶음
1. LangChain과 FAISS로 RAG를 설계할 때 주요 구성요소(임베딩·인덱싱·검색·생성
   연동)와 역할을 설명하고, 신규 이력서 추가 시 인덱스를 실시간 업데이트하는
   방법을 어떻게 구현하겠는가? (← 여러 주제 혼합)
2. 멀티에이전트 역할 분담과 통신 방식, 세션 컨텍스트 관리를 모두 설명하라.
3. FAISS 임베딩 모델·차원·유사도 선택 이유와 환각 감소 전략을 함께 제안하라.

### V2 (프롬프트) — 직무·근거는 좋아졌으나 여전히 복합 질문
1. RAG/FAISS 구현 흐름(임베딩·분할·인덱스·유사도·Top-K)과 관련성 개선 조치를
   모두 설명하라. (← 한 문장에 6개)
2. 에이전트 역할 분담 + 컨텍스트 공유 + 토큰 한계 처리까지 한 번에.
3. 배포 후 모니터링 + 실패 사례 + 개선 실험을 한꺼번에.

### V3 (+단일주제) — 구체적이고 한 번에 하나씩 ✅
1. RAG 파이프라인에서 어떤 임베딩 모델을 썼고 선택 이유는 무엇인가요?
2. FAISS 인덱스 구축 시 문서 분할(chunking) 전략을 어떻게 설계했나요?
3. RAG 응답의 환각(hallucination)을 어떻게 검증하고 개선하려 했나요?
